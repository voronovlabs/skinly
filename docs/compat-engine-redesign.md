## 1. Общая оценка текущего движка

Каркас правильный и его стоит сохранить: декларативные правила, эмитящие
типизированные `RuleHit` (kind/weight/inci/kbId), чистая агрегация в
`score.ts`, отделённая explainability, pure-функция без I/O, DM-словарь
как источник семантики. Это ровно та архитектура, на которой можно
масштабировать качество данными, а не кодом.

Проблема не в каркасе, а в **модели агрегации**: линейная сумма
per-ingredient весов поверх щедрого baseline 75. Из неё механически следуют
все главные аномалии аудита:

| Сигнал аудита | Значение |
|---|---|
| excellent | 27.1% всех оценок (852/3146), из них 380 — ровно 100 |
| excellent при warnings > 0 | 206 случаев (B1) |
| score ≥ 85 только на generic-увлажнителях | 231 случай (B3) |
| concern acne меняет score | у 14/121 товаров, средний Δ **+0.7** |
| goal не влияет и не упоминается | 277 случаев (A1) |
| товары, где concerns/goals не меняют НИЧЕГО | 22.3% |
| дубли в reasons (`dryFriendly×4`) | 1752 случая; ≥4 повторов одного ключа — в 798/3146 оценок |
| reasons пустые | 891 случай (все — normal/combination кожа) |
| warning не попадает в топ-reasons | 570 случаев |
| avoided-канал | работает: только 14 C2, verdict корректно падает |

## 2. Главные проблемы

### P1. Инфляция generic-позитивов (фундаментальная)

`skin_dry` даёт +5 **за каждый** humectant/barrier/occlusive. В креме
таких 8–27 штук. Diminishing returns начинается только после +30 — то есть
6 увлажнителей уже гарантируют потолок. В dry-сценариях 111/363 оценок
насыщают dampener одними увлажнителями. Отсюда score=100, verdict=excellent
у продукта «просто крем» — и у аудитного кейса 3700203746817 (dry/none →
100 при 8×dryFriendly).

Это фундаментально: **сигнал пропорционален длине состава, а не качеству
подбора**. Любая длинная формула с водой-глицерином-маслами обыгрывает
точечный актив.

### P2. Персонализация задавлена (фундаментальная)

Concern-правило (+8..12 за ингредиент) и goal (+4) добавляются к сумме,
которая уже упёрлась в dampener/потолок 100 → добавка исчезает
(Δ(c_dry_acne)=+0.7 в среднем). Пользователь меняет анкету — результат не
меняется. Это ровно то «отсутствие персонализации», которое видно в
наблюдении: oily+oilControl+acne даёт те же 89/excellent, что и oily
без всего.

Вторая причина — **данные**: `helpsConcern` сработал 370 раз против 2945
`dryFriendly` → `benefits_for` в `dm.ingredient_properties` заполнен
редко. Правило не может сработать на пустой разметке.

### P3. Verdict игнорирует warnings (фундаментальная для доверия)

`pickVerdict` — чистые пороги по score (кроме avoided). Один `oilyHeavy`
(−10) арифметически тонет в стопке позитивов → «excellent, но с
предупреждением о комедогенном масле» (206 случаев). Для пользователя это
прямое противоречие: *продукт «идеально подходит» и одновременно «может
забивать поры»*.

### P4. Baseline 75 = «всё по умолчанию хорошее»

86% оценок — good/excellent. Шкала не различает «нормально» и «отлично».
Дополнительно `lowConfidence` подтягивает нераспознанные составы **вверх к
75** («неизвестно» = «good») — 24.8% оценок lowConfidence, и все они
выглядят благополучно.

### P5. Explainability повторяет один ключ и молчит о рисках (вторичная, но самая видимая)

`pickTopReasons` берёт топ-4 по weight — а это 4 одинаковых `dryFriendly`.
Warnings попадают в reasons только если позитивов < 2 (570 случаев C3 —
предупреждение есть, в объяснении его нет). Для normal-кожи правил почти
нет → 891 пустое объяснение.

### P6. Разрыв словаря анкета ↔ движок (вторичная, дешёвая)

`dryness`, `oiliness`, `oilControl`, `soothing`, `brightening`,
`silicones` — тихий no-op (probes подтвердили). Пользователь думает, что
профиль учтён.

### Дополнительные архитектурные проблемы (вне выводов аудита)

- **Тройной учёт одного ингредиента.** Глицерин у dry+hydration
  собирает +5 (skin_dry) и +4 (goal) — а на combination ещё и
  combinationBalanced. Каналы не изолированы; g_dry_hydration — самый
  высокий средний score выборки (89.5) именно поэтому.
- **Позиция в составе не используется.** `IngredientFact.position` есть,
  но 25-й ингредиент весит как 1-й. Ароматизатор в хвосте и актив в топ-3
  неразличимы.
- **Числовые risk-поля не используются.** `irritancy/comedogenicity/
  allergenicity` (0–3) есть в properties и в фактах, но sensitivity-правило
  смотрит только теги. Ингредиент с irritancy=3 без «нужного» тега невидим.
- **Пороги смежных систем зашиты под текущую шкалу.** Recommendations
  (COMPAT_GATE 60/50, classifyRecommendation 75/60) и ScanHistory.matchScore
  калиброваны под инфлированное распределение — любое изменение шкалы
  требует их синхронной перекалибровки и версии алгоритма.
- **`weight` смешивает «силу сигнала» и «вклад в score».** Нельзя поднять
  важность warning'а в explainability, не изменив арифметику.

Что ведёт к **неправильным рекомендациям** (не только к плохим текстам):
P1+P4 (завышенные score попадают в reco-gate и ранжирование), P3 (excellent
при комедогенности для oily/acne), P2 (два разных пользователя получают
один ответ), P6 (профиль игнорируется). Только на **explainability** влияют
P5 и часть C3/D1/D2.

## 3. Что уже сделано хорошо

- Rules → hits → aggregation: расширяемо, тестируемо; правила не двигают
  score напрямую.
- Avoided-канал: hard-penalty −25 + cap 60 + verdict-даунгрейд. Аудит
  подтверждает (fragrance: 56→25 risky; C1=0, C2=14 — почти все из-за
  неполной разметки `flags_avoided`, не из-за модели).
- Sentinel score=0 для пустого профиля; floor 25 (не пугаем нулями).
- Sensitivity-канал заметен (a_dry_high 56 против a_dry_none 100 на
  спорном товаре — направление верное).
- Pure/deterministic, O(rules×facts), i18n-ключи с аргументами.
- DM-словарь отделён от кода — качество наращивается данными.

## 4. Что необходимо изменить (сводка решений)

1. **Заменить линейную сумму на канальную агрегацию с caps** —
   полная замена внутренностей `score.ts` (это то самое «решение, которое
   стоит заменить целиком»: линейная сумма поверх baseline 75 не лечится
   подкруткой весов — любой вес инфлируется длиной состава).
2. **Насыщение per-rule по числу РАЗНЫХ ингредиентов** вместо
   per-ingredient суммы (лечит P1).
3. **Поднять канал персонализации и дать ему собственный бюджет,
   превышающий generic-fit** (лечит P2).
4. **Verdict: severity-модель + gating**, а не только пороги score
   (лечит P3).
5. **Baseline 62 + «неизвестное ≠ хорошее»**: lowConfidence сжимает к 62,
   а не к 75; уверенность — отдельный выход (лечит P4).
6. **Explainability: агрегация повторов, гарантированное упоминание
   warnings, fallback-причина для нейтральных составов** (лечит P5).
7. **Словарный адаптер анкеты** (лечит P6) + план разметки
   `benefits_for`/`flags_avoided`.

## 5. Предлагаемая новая модель scoring

### 5.1. Структура: четыре канала вместо одной суммы

Правила продолжают эмитить hits, но каждый hit получает `channel` и
`severity`; агрегация — по каналам, у каждого свой бюджет (cap):

```
score = clamp( BASE
             + FIT            // generic-совместимость с типом кожи   ≤ +12
             + PERSONAL       // concerns + goal                       ≤ +26
             + RISK,          // все штрафы                            ≥ −45
             15, 100 )
BASE = 62
```

Ключевое следствие: PERSONAL (до +26) **доминирует** над FIT (до +12) —
пользователь с заполненной анкетой всегда чувствует её влияние, а «просто
хороший крем» без попадания в concerns физически не может подняться выше
62+12 = 74 (good, не excellent). Excellent становится достижим только через
персональное попадание — это и есть требуемая персонализация без
нестабильности: бюджеты каналов жёстко ограничивают амплитуду.

### 5.2. FIT — насыщение по разнообразию, а не сумме

Для каждого skin-type-правила считаем **число различных ингредиентов** n,
попавших в правило, и применяем вогнутую кривую:

```
fitRaw = Σ_group  cap_group × (1 − exp(−n_group / k_group))
FIT = min(12, fitRaw)

группы (для dry): humectant/barrier/occlusive  cap=10, k=4
                  soothing                      cap=3,  k=2
(для oily):       light humectant               cap=8,  k=4
(для combination/normal): аналогично, cap 6–8
```

Свойства: 1 увлажнитель ≈ +2.2, 4 ≈ +6.3, 8 ≈ +8.6, 27 ≈ +10 — рост есть,
но 6-й ингредиент больше не «делает excellent». Один и тот же ингредиент
учитывается в FIT **один раз** (по самой сильной группе) — конец
тройному учёту.

### 5.3. PERSONAL — бюджет на concern с покрытием

```
для каждого concern из профиля:
  best  = сильнейший benefits_for-ингредиент: active +10 / другой +6
  extra = каждый следующий РАЗНЫЙ ингредиент +3, но не более +6
  perConcern = min(16, best + extra)

concernScore = Σ perConcern, но не более 20
goalScore    = min(8, аналогичная кривая по goal-match: best +5, extra +1.5)
PERSONAL     = min(26, concernScore + goalScore)
```

- `cautionsFor` по concern уходит в RISK (не взаимозачитывается с
  позитивами внутри канала).
- Если concern заявлен, а benefits-ингредиентов нет → эмитится
  **info-hit `concernNotCovered`** (не штраф, но explainability и триггер
  для verdict-gate excellent, см. §6).
- Goal, дублирующий concern (hydration при dry), не суммируется с FIT за
  тот же ингредиент: ингредиент приписывается к каналу с большим вкладом
  (PERSONAL > FIT), второй канал его не видит.

Стабильность: изменение одного пункта анкеты меняет score максимум на
величину его бюджета (concern ≤ 16, goal ≤ 8, avoided ≤ полный RISK) —
предсказуемо и монотонно; аудит-инварианты E1–E4 выполняются по построению.

### 5.4. RISK — severity-модель с внутригрупповым затуханием

Каждый warning получает severity:

| Severity | Примеры | Базовый вес |
|---|---|---|
| **blocker** | avoidedFlag (нарушение явного запрета); в будущем — аллерген/беременность+ретиноид | −25 первый, −8 каждый следующий РАЗНЫЙ флагнутый ингредиент |
| **major** | sensitiveTrigger (high/reactive), cautionForConcern, oilyHeavy, dryStripping | reactive −18 / high −12 / прочие −10 |
| **minor** | activeForSensitive, combinationComedogenic | −6 |

Агрегация внутри одной (severity, key)-группы — затухание 1 / 0.5 / 0.25 /
0.25… по РАЗНЫМ ингредиентам (5 отдушек — не 5×(−12)=−60, а −21: риск
растёт, но не линейно). Числовые поля включаются в модель: ингредиент с
`irritancy ≥ 2` при sensitivity=high добавляет major-hit, даже если тега
нет (задействуем уже существующие данные). Итог:

```
RISK = −min(45, Σ groupDecayedPenalties)
```

Hard-cap avoided (60) сохраняется как страховка поверх каналов.

### 5.5. Confidence — отдельный выход, сжатие вниз

```
confidence = high (recognized ≥ 0.6 и facts ≥ 8)
           | medium (0.3–0.6)
           | low (< 0.3)
low → score = round(0.6×score + 0.4×62)   // к НЕЙТРАЛИ, не к 75
```

«Неизвестный состав» перестаёт выглядеть как «хороший» — он выглядит как
«непонятный» (mixed-зона + плашка lowConfidence). `confidence` добавляется
в `CompatibilityResult` (аддитивно, контракт не ломается).

### 5.6. Position-aware вес (Phase 2)

Позиция как прокси концентрации, только для позитивов:
топ-треть состава ×1.0, середина ×0.7, хвост ×0.4. Штрафы позицию
игнорируют (отдушка в хвосте — всё равно отдушка для аллергика).

## 6. Предлагаемая модель verdict

Verdict = пороги score **после** гейтов (порядок важен):

```
1. есть blocker            → максимум mixed; risky, если score < 45 или blockers ≥ 2
2. есть major-warning      → максимум good      (excellent недостижим — конец B1)
3. excellent требует ВСЕ:  score ≥ 85
                           ∧ ни одного major/blocker
                           ∧ ≥1 PERSONAL-позитив (helpsConcern/goalAlignment)
                             ЛИБО профиль без concerns/goals
                           ∧ confidence ≠ low
                           ∧ нет concernNotCovered ни по одному заявленному concern
4. пороги: ≥85 excellent · 70–84 good · 45–69 mixed · <45 risky
```

Целевое распределение на текущем каталоге (проверяется re-аудитом):
excellent ~5–10%, good ~30–40%, mixed ~40–50%, risky ~8–12%. Это резкий
сдвиг относительно нынешних 27/59/10/4 — и он честный: mixed = «обычный
нормальный продукт без персонального попадания».

## 7. Предлагаемая модель explainability

1. **Агрегация повторов на уровне результата.** Hits группируются по
   (key, concern/avoided); группа рендерится ОДНОЙ причиной с количеством и
   примерами: «8 увлажняющих компонентов для сухой кожи — глицерин,
   гиалуронат, сквалан и ещё 5». Новые i18n-ключи `*Many` с `{count}`,
   `{examples}` (RU/EN). Это убивает `dryFriendly×4` (1752 случая D2) и
   делает reasons короче и информативнее.
2. **Гарантированный состав топ-reasons** (фикс C3): слоты
   `[blocker?] + [топ PERSONAL-позитив?] + [топ warning, если verdict<excellent] +
   [топ FIT-позитив]` — предупреждение не может исчезнуть из объяснения,
   если оно повлияло на verdict.
3. **Fallback-причины** (фикс D1, 891 случай): нет hits →
   `neutralComposition` («состав нейтрален: без отдушек, агрессивных ПАВ и
   спиртов» — если это правда по фактам) или `lowCoverage` при
   lowConfidence.
4. **Честность персонализации:** `concernNotCovered` («активов под акне в
   составе нет») — пользователь видит, что анкета УЧТЕНА, даже когда
   совпадений нет. Это explainability-ответ на P2 не менее важный, чем веса.
5. `weight` в RuleHit разделяется на `impact` (вклад в score) и
   `salience` (приоритет в UI) — предупреждения можно показывать первыми,
   не трогая арифметику.

## 8. Новые веса и caps (сводная таблица)

| Параметр | Сейчас | Предлагается |
|---|---|---|
| BASELINE | 75 | **62** |
| Floor/Ceil | 25/100 | **15**/100 |
| FIT cap | нет (dampener после +30) | **+12**, кривая 1−exp(−n/4) на РАЗНЫХ ингредиентах |
| PERSONAL cap | нет отдельного | **+26** (concern ≤16 каждый, Σ concerns ≤20, goal ≤8) |
| concern best-hit | +8..12 линейно за каждый | **+10 active / +6**, extra +3 (≤ +6) |
| goal hit | +4 за каждый | best **+5**, extra +1.5, cap +8 |
| RISK cap | нет (только clamp) | **−45**, внутригрупповое затухание 1/0.5/0.25 |
| avoidedFlag | −25 каждый | −25 первый, −8 следующие РАЗНЫЕ; blocker-gate verdict |
| sensitiveTrigger | −12/−18 каждый | −12/−18 первый в группе, затухание |
| lowConfidence | сжатие к 75 | сжатие к **62** + `confidence` в результате |
| verdict | пороги 88/72/50 | гейты (§6) + пороги **85/70/45** |
| dampener +30 → ×0.5 | есть | **удаляется** (заменён канальными caps) |

Пример на кейсе 3700203746817 (увлажняющий, 1 heavy-oil, отдушка):
dry/none: 62+10.4(8 увл.)+0−0 = **72 good** (было 100 excellent);
dry+acne без анти-акне активов: 72 + `concernNotCovered` → **good**, не
excellent (было 100); oily/none: 62+8.6−10 = **61 mixed** с явной причиной
про масло (было 89 excellent при том же warning); dry/high+avoid fragrance:
62+10.4−(25+12+…) ≈ **25–35 risky** (как сейчас — канал работал).

## 9. Приоритетный roadmap

### Phase 1 — максимум эффекта, минимум кода (score.ts/rules.ts/explain, без схемы БД)

1. Verdict-гейты (§6) — убивает B1/B2 полностью (206 случаев).
2. Групповое насыщение FIT-правил + удаление dampener + BASELINE 62 —
   убивает P1/P4 (B3: 231; сжимает 380 «сотен»).
3. Бюджеты PERSONAL с новыми весами — лечит A1-механику (веса).
4. Explainability: агрегация повторов, слоты reasons, fallback-причины,
   `concernNotCovered` — D1/D2/C3 (3213 случаев суммарно).
5. Словарный адаптер: `dryness/oiliness → skinType`, `oilControl →
   clear_skin`, `brightening → even_tone`, `soothing → redness`,
   `silicones` → новый flag/тег (KB) или явное «не поддерживается».
6. lowConfidence → сжатие к 62 + поле confidence.
7. **Регрессия:** `npm run audit:compat` тем же seed до/после + golden-set
   юнит-тесты (кейс 3700203746817 и топ-20 аудита) + сверка целевого
   распределения verdicts. Синхронно перекалибровать
   `COMPAT_GATE 60→55`, `classifyRecommendation 75/60 → 78/62` в reco
   (проверить re-bench'ем) и пометить результат `algoVersion: 2`
   (ScanHistory-снэпшоты со старой шкалой не пересчитываем).

### Phase 2 — средний объём

1. Канальный рефакторинг `score.ts` в явные ChannelScore-структуры
   (FIT/PERSONAL/RISK как first-class объекты в результате — для UI
   breakdown и отладки).
2. Position-aware веса позитивов (×1.0/0.7/0.4).
3. Numeric-risk: irritancy/comedogenicity/allergenicity из properties в
   RISK-канал (sensitivity перестаёт зависеть только от тегов).
4. Avoided-severity: два уровня (`preference` — текущее поведение;
   `declared_allergy` — hard fail: verdict risky, score ≤ 40) — поле в
   профиле, аддитивная миграция.
5. **Данные:** разметка `benefits_for`/`flags_avoided`/тегов для топ-200
   canonical-ингредиентов по частоте (сейчас helpsConcern физически не
   может сработать на большинстве составов — это главный长-term рычаг
   персонализации, не код).
6. Аудит в CI: пороговые проверки (share excellent ≤ 12%, B1=0, D2=0…).

### Phase 3 — серьёзный рефакторинг

1. Расширение профиля (беременность/аллергены/ретиноиды из онбординга —
   сейчас выбрасываются, см. recommender-audit §1.1) + safety-blockers.
2. Interaction graph (retinol+AHA, vit C+niacinamide…) как новые
   RISK-правила (Phase 10.4 плана проекта).
3. `benefit_strength` (0–3) в ingredient_properties — сила ингредиента
   против concern вместо бинарного benefits_for; веса PERSONAL начинают
   читать силу.
4. Калибровка весов по телеметрии (UserProductEvent / recommendation_logs)
   — grid-search по детерминированным параметрам, БЕЗ ML в рантайме.

## 10. Итоговая архитектура движка

```
IngredientFact[] + CompatibilityProfile
        │
   RULES (декларативные, как сейчас) → RuleHit[]
        │   + channel: fit | personal | risk | info
        │   + severity: blocker | major | minor | none
        │   + impact / salience (вклад ≠ приоритет показа)
        ▼
   ChannelAggregator
     FIT      = sat-кривые по РАЗНЫМ ингредиентам, cap +12
     PERSONAL = бюджеты concern/goal, cap +26   ← доминирует над FIT
     RISK     = severity + внутригрупповое затухание, cap −45
     (ингредиент приписан одному каналу: personal > fit)
        ▼
   score = clamp(62 + FIT + PERSONAL + RISK, 15, 100)
   confidence = f(recognizedRatio, facts) ; low → сжатие к 62
        ▼
   VerdictGate (blockers → ≤mixed; majors → ≤good;
                excellent требует персонального попадания и не-low confidence)
        ▼
   Explain: группировка hits → агрегированные причины с count/примерами,
            слоты [blocker | personal | warning | fit], fallback-причины,
            concernNotCovered
        ▼
   CompatibilityResult (контракт прежний + confidence, channels, algoVersion)
```

Всё остаётся pure/deterministic, O(rules×facts), без внешних вызовов;
API `/products/:id/compatibility`, web-секция и mobile-карточка получают
изменения только аддитивно. Регрессионная петля: детерминированный
audit:compat (тот же seed) + golden-тесты + распределения — до и после
каждой фазы.
